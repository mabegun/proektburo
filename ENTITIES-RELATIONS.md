## 1. Сущности

### 1.1 Project (Проект)

**Назначение:** Основная единица учёта — проект документации

| Поле | Тип | Обяз. | Описание |
|------|-----|-------|----------|
| id | UUID | ✅ | Уникальный идентификатор |
| code | String | ✅ | Шифр проекта (2025-001) |
| name | String | ✅ | Название проекта |
| status | Enum | ✅ | Статус (draft, in_progress, on_approval, completed, archived) |
| deadline | Date | | Контрольная дата |
| inputText | Text | | Текстовое описание ТЗ |
| inputFiles | File[] | | Файлы ТЗ, ГП участка |
| createdBy | Employee | ✅ | ГИП проекта (автор) |
| createdAt | DateTime | ✅ | Дата создания |
| updatedAt | DateTime | ✅ | Дата обновления |

**Связи:**
- `sections: Section[]` — разделы проекта (1:N)
- `surveys: Survey[]` — изыскания проекта (1:N)
- `expenses: Expense[]` — расходы проекта (1:N)
- `comments: Comment[]` — обсуждение проекта (1:N)
- `expertiseComments: ExpertiseComment[]` — замечания экспертизы (1:N)

**Страницы:** `03-projects-list.html`, `04-project-detail.html`, `04-1-project-detail-employee.html`, `05-project-create.html`

**UI-особенности:**
- **Для ГИП/Директора** (`04-project-detail.html`): видны все разделы единым списком, блок "Команда проекта" доступен
- **Для Исполнителя** (`04-1-project-detail-employee.html`): разделы разделены на группы:
  - «Мои разделы» — где пользователь ответственный или соисполнитель
  - «Доступны для просмотра» — где пользователь наблюдатель (👁)
  - Блок "Команда проекта" скрыт
  - Изыскания показываются только те, где пользователь в observers

| Удаление файлов | ✅ | ✅ | ✅* | ❌ |

| Финансы | ✅ | ✅ | ❌ | ❌ |
| История (полная) | ✅ | ✅ | ⚠️ Краткая | ⚠️ Краткая |

### 1.3 Survey (Изыскание)

**Назначение:** Инженерное изыскание по проекту

| Поле | Тип | Обяз. | Описание |
|------|-----|-------|----------|
| id | UUID | ✅ | Уникальный идентификатор |
| projectId | UUID | ✅ | Ссылка на проект |
| surveyTypeId | UUID | ✅ | Ссылка на вид изыскания (справочник) |
| status | Enum | ✅ | Статус (аналогично Section) |
| responsible | Employee | ✅ | Ответственный от организации |
| contractor | Contractor | | Внешний исполнитель (юрлицо) |
| contractFile | File | | Файл договора |
| deadline | Date | | Дедлайн |
| createdAt | DateTime | ✅ | Дата создания |

**Связи:**
- `project: Project` — проект (N:1)
- `surveyType: SurveyType` — вид изыскания из справочника (N:1)
- `files: File[]` — файлы изыскания (1:N)
- `comments: Comment[]` — обсуждение (1:N)

**Страницы:** `04-project-detail.html`, `14-survey-detail.html` ✅

**UI-особенности:**
- На странице проекта в блоке изысканий есть кнопка быстрого добавления "+ Добавить"
- Индикаторы изысканий на карточках проектов используют CSS-класс `.survey-indicator`
- Tooltips с информацией об изыскании используют CSS-класс `.survey-tooltip`
- **Индикаторы изысканий для исполнителя (IDEAS.md 2.4 ✅):**
  - Исполнитель видит изыскания, где он в `observers`
  - Индикаторы помечены меткой `И:` перед разделами
  - В tooltip указана роль "👁 Наблюдатель"

**Наблюдатели изыскания:**
| Поле | Тип | Обяз. | Описание |
|------|-----|-------|----------|
| observers | Employee[] | | Участники проекта, которым открыт просмотр изыскания |

**Правила доступа к изысканию:**
| Роль | Видит изыскание |
|------|----------------|
| Директор | Все изыскания |
| ГИП | Все изыскания |
| Исполнитель | Только если в observers |

**Матрица видимости страницы изыскания (для observers):**

| Блок | Директор/ГИП | Исполнитель (observer) |
|------|:------------:|:----------------------:|
| Статус, дедлайн, ответственный | ✅ Полный | ✅ Полный |
| Вид изыскания | ✅ | ✅ |
| **Блок "Исполнитель (юрлицо)"** | ✅ Полный | ❌ **Полностью скрыт** |
| **Договор (файл)** | ✅ | ❌ **Полностью скрыт** |
| **Файлы изыскания** | ✅ Полный | ⚠️ Только просмотр |
| — Скачивание | ✅ | ✅ |
| — Загрузка/удаление | ✅ | ❌ |
| — Согласование | ✅ | ❌ |
| **Блок "Кто видит изыскание"** | ✅ | ❌ Полностью скрыт |
| **Обсуждение** | ✅ Полный | ✅ Полный |
| — Написать сообщение | ✅ | ✅ |
| **История изменений** | ✅ Полная | ⚠️ Краткая |
| **Кнопка "Редактировать"** | ✅ | ❌ |
| **Кнопка "Назад к проекту"** | — | ✅ Добавлена |

**Причины ограничений для observers:**
- Финансовая информация (договоры, суммы) — коммерческая тайна
- Контактные данные подрядчиков — защита от утечки
- Исполнители получают доступ только для ознакомления с результатами изысканий

### 1.16 ProjectMember (Участник проекта)

**Назначение:** Связующая сущность для хранения команды проекта — списка участников, которые участвуют в проекте.

| Поле | Тип | Обяз. | Описание |
|------|-----|-------|----------|
| id | UUID | ✅ | Уникальный идентификатор |
| projectId | UUID | ✅ | Ссылка на проект |
| employeeId | UUID | ✅ | Ссылка на исполнителя |
| roleInProject | String | | Роль в проекте (ГИП, Архитектор, Инженер ОВ...) |
| addedBy | UUID | | Кто добавил |
| addedAt | DateTime | ✅ | Дата добавления |

**Ограничения:**
- Уникальная пара (projectId, employeeId) — исполнитель может быть в команде проекта только один раз

**Связи:**
- `project: Project` — проект (N:1)
- `employee: Employee` — исполнитель (N:1)

**Страницы:** `04-project-detail.html` — блок "Команда проекта"

**Где используется:**
| Место | Назначение |
|-------|------------|
| Выбор соисполнителей в разделе | Выпадающий список из project_members |
| Выбор наблюдателей в разделе | Выпадающий список из project_members |
| Выбор observers в изыскании | Выпадающий список из project_members |
| Индикаторы разделов у исполнителя | Фильтрация по project_members |

**Автоматическое добавление:**
- При создании проекта → ГИП добавляется в команду
- При назначении ответственного за раздел → добавляется в команду (если нет)

## 7. SQL-схемы (справочно)

### Основные таблицы

```sql
-- Проекты
CREATE TABLE projects (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  code VARCHAR(20) UNIQUE NOT NULL,
  name VARCHAR(255) NOT NULL,
  status VARCHAR(50) DEFAULT 'draft',
  deadline DATE,
  input_text TEXT,                    -- Вводные данные (ТЗ)
  created_by UUID REFERENCES employees(id),
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP DEFAULT NOW()
);

-- Разделы проекта
CREATE TABLE sections (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  project_id UUID REFERENCES projects(id) NOT NULL,
  section_type_id UUID REFERENCES section_types(id) NOT NULL,
  status VARCHAR(50) DEFAULT 'not_started',
  responsible_id UUID REFERENCES employees(id),
  deadline DATE,
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP DEFAULT NOW()
);

-- Изыскания
CREATE TABLE surveys (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  project_id UUID REFERENCES projects(id) NOT NULL,
  survey_type_id UUID REFERENCES survey_types(id) NOT NULL,
  status VARCHAR(50) DEFAULT 'not_started',
  responsible_id UUID REFERENCES employees(id),
  contractor_id UUID REFERENCES contractors(id),
  deadline DATE,
  created_at TIMESTAMP DEFAULT NOW()
);

-- Расходы
CREATE TABLE expenses (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  project_id UUID REFERENCES projects(id) NOT NULL,
  section_id UUID REFERENCES sections(id),
  category_id UUID REFERENCES expense_categories(id) NOT NULL,
  amount DECIMAL(12,2) NOT NULL,
  description TEXT,
  status VARCHAR(50) DEFAULT 'pending',
  date DATE NOT NULL,
  created_by UUID REFERENCES employees(id),
  approved_by UUID REFERENCES employees(id),
  approved_at TIMESTAMP,
  created_at TIMESTAMP DEFAULT NOW()
);

-- Исполнители
CREATE TABLE employees (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  first_name VARCHAR(100) NOT NULL,
  last_name VARCHAR(100) NOT NULL,
  middle_name VARCHAR(100),
  position VARCHAR(100) NOT NULL,
  role VARCHAR(50) NOT NULL,
  email VARCHAR(255) UNIQUE NOT NULL,
  phone VARCHAR(20),
  avatar_color VARCHAR(7),
  is_active BOOLEAN DEFAULT true,
  created_at TIMESTAMP DEFAULT NOW()
);

-- Компетенции исполнителей
CREATE TABLE employee_competencies (
  employee_id UUID REFERENCES employees(id),
  section_type_id UUID REFERENCES section_types(id),
  PRIMARY KEY (employee_id, section_type_id)
);

-- Команда проекта (участники)
CREATE TABLE project_members (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  project_id UUID REFERENCES projects(id) NOT NULL,
  employee_id UUID REFERENCES employees(id) NOT NULL,
  role_in_project VARCHAR(100),              -- Роль в проекте (ГИП, Архитектор...)
  added_by UUID REFERENCES employees(id),    -- Кто добавил
  added_at TIMESTAMP DEFAULT NOW(),
  UNIQUE(project_id, employee_id)            -- Один исполнитель — одна запись в проекте
);

-- Наблюдатели изысканий (кому открыт просмотр)
CREATE TABLE survey_observers (
  survey_id UUID REFERENCES surveys(id),
  employee_id UUID REFERENCES employees(id),
  added_by UUID REFERENCES employees(id),
  added_at TIMESTAMP DEFAULT NOW(),
  PRIMARY KEY (survey_id, employee_id)
);
```

### 8.1 Список проектов с индикаторами разделов (`03-projects-list.html`)

Для отображения карточек проектов с индикаторами разделов:

```sql
-- Получить список проектов с разделами для индикаторов
SELECT 
  p.id,
  p.code,
  p.name,
  p.status,
  p.deadline,
  -- Разделы проекта для индикаторов (JSON-агрегация)
  COALESCE(
    json_agg(
      json_build_object(
        'shortName', st.short_name,
        'fullName', st.full_name,
        'color', st.color,
        'status', s.status
      )
      ORDER BY st.short_name
    ) FILTER (WHERE s.id IS NOT NULL),
    '[]'::json
  ) as sections,
  -- Подсчёт статистики
  COUNT(s.id) as sections_count,
  COUNT(e.id) as employees_count,
  -- Прогресс
  ROUND(
    100.0 * COUNT(CASE WHEN s.status = 'approved' THEN 1 END) / 
    NULLIF(COUNT(s.id), 0)
  ) as progress_percent
FROM projects p
LEFT JOIN sections s ON s.project_id = p.id
LEFT JOIN section_types st ON s.section_type_id = st.id
LEFT JOIN employees e ON e.id = s.responsible_id
WHERE p.status != 'archived'
GROUP BY p.id
ORDER BY p.created_at DESC;
```

### 8.1.1 Список проектов для исполнителя (ролевая фильтрация)

Для исполнителей показываются только проекты с доступными разделами:

```sql
-- Получить проекты, доступные исполнителю (с ролевой фильтрацией)
SELECT DISTINCT
  p.id,
  p.code,
  p.name,
  p.status,
  p.deadline,
  -- Разделы с указанием типа доступа для текущего пользователя
  COALESCE(
    json_agg(
      json_build_object(
        'id', s.id,
        'shortName', st.short_name,
        'fullName', st.full_name,
        'status', s.status,
        'accessType', 
          CASE 
            WHEN s.responsible_id = :currentUserId THEN 'responsible'
            WHEN :currentUserId = ANY(s.co_executors) THEN 'co_executor'
            WHEN :currentUserId = ANY(s.observers) THEN 'observer'
            ELSE NULL
          END
      )
      ORDER BY st.short_name
    ) FILTER (WHERE s.id IS NOT NULL AND (
      s.responsible_id = :currentUserId OR
      :currentUserId = ANY(s.co_executors) OR
      :currentUserId = ANY(s.observers)
    )),
    '[]'::json
  ) as accessible_sections
FROM projects p
JOIN sections s ON s.project_id = p.id
JOIN section_types st ON s.section_type_id = st.id
WHERE p.status != 'archived'
  -- Фильтр: есть доступ хотя бы к одному разделу
  AND EXISTS (
    SELECT 1 FROM sections s2
    WHERE s2.project_id = p.id
    AND (
      s2.responsible_id = :currentUserId OR
      :currentUserId = ANY(s2.co_executors) OR
      :currentUserId = ANY(s2.observers)
    )
  )
GROUP BY p.id
ORDER BY p.created_at DESC;
```
